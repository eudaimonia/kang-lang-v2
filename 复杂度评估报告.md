# Kang v2 复杂度评估报告

## 项目定位

Kang 是一门教学用 toy 语言，严格静态类型，支持 JIT REPL 和 AOT 编译到二进制。
实现语言 Rust，技术栈 inkwell + logos + ariadne，运行时静态链接 musl libc。

## 模块分解与估量

### 1. 词法分析 — 低复杂度 (~200 LOC)

- 约 30 个 token 类型：关键字（def/var/return/if/else/then/for/in/struct）、字面量（i32/f64/str/bool）、运算符、分隔符
- logos 声明式词法，正则匹配 + 优先级，几乎不需要手写状态机
- 需要处理注释（`//` 行注释，`/* */` 块注释）和字符串转义（`\n \t \\ \" \0`）

### 2. 语法分析 — 中复杂度 (~900 LOC)

- 递归下降手写 parser，LL(1) 文法无歧义
- 表达式用 Pratt parsing 处理优先级（一元负号、乘除、加减、比较、逻辑）
- 语句边界清晰（块/变量/赋值/返回/分支/循环/结构体/函数调用）
- 结构体定义限制顶层，简化作用域管理
- 二值返回：ReturnType 解析 `(T1, T2)`，var 多接收绑定，return 逗号分隔表达式
- 易错点：for 循环的 init/cond/step 三部分解析、结构体构造 vs 函数调用的区分、`_` discard 无类型标注的特殊处理

### 3. AST + 类型表示 — 低复杂度 (~350 LOC)

- 表达式节点：Literal、Variable、Binary、Unary、Call、Index、Field、Grouping
- 语句节点：Block、VarDecl、Assign、Return、If、For、StructDef、ExprStmt
- 类型表示：I32、F64、Str、Bool、Void、Array(T)、Struct(name, fields)
- 函数签名：Params + ReturnType（单类型或二值 tuple）
- VarDecl 支持单/双绑定，Binding 含可选类型（`_` 无类型）
- 均实现 Copy/Clone，值语义

### 4. 语义分析与类型检查 — 中复杂度 (~900 LOC)

检查规则清单（约 30 条）：

**类型规则**
1. 变量/参数必须显式标注类型（`_` discard 除外）
2. 禁止隐式类型转换（str + 例外）
3. 算术/比较运算仅限同类型（i32-i32 或 f64-f64，不可混用）
4. 索引必须是 i32
5. 条件必须是 bool
6. 等值比较仅限同类型
7. 结构体构造必须全字段初始化
8. 禁止 `[void]` 和 void 结构体字段

**二值返回**
9. 函数返回类型必须与 return 表达式数量/类型一致（单值或二值）
10. var 接收变量数（1 或 2）必须与函数返回值数一致，单接收取第一值
11. `_` 仅出现在 var 多接收中，不作为变量绑定
12. `(T1, T2)` 类型不可出现在 var 声明/参数/结构体字段中（语法层已杜绝）

**作用域与命名**
13. 函数名与变量名共享命名空间，同名冲突报错
14. 函数参数不可在体内被 var 重新声明
15. 内层块变量可遮蔽外层
16. 变量使用前在作用域内已声明

**流程控制**
17. 非 void 函数所有代码路径必须显式 return
18. void 函数只能用 `return;`（无表达式）
19. 结构体定义仅限顶层

**赋值**
20. str 不可变，`s[i]` 禁止作为左值
21. 赋值左右类型必须匹配
22. 结构体字段访问的类型推断

**结构体**
23. 字段类型禁止 void
24. 禁止直接或间接自引用（仅允许通过 `[Self]` 间接引用）

**其他**
25. 函数重载仅限内置函数
26. `bool(s: str)` 仅接受精确 `"true"` / `"false"`（小写）

### 5. LLVM IR 代码生成 — 高复杂度 (~1,800 LOC)

**类型映射**
| Kang | LLVM |
|------|------|
| i32 | i32 |
| f64 | double |
| bool | i1 |
| void | void |
| str | { i8*, i64 }（指针 + 长度） |
| [T] | { T*, i64, i64 }（指针 + 长度 + 容量） |
| struct | 对应 LLVM struct type |
| (T1, T2) | 结构体 { T1, T2 } 或分别返回 (依 ABI) |

**核心挑战**
- **SSA 构造**：表达式树到 LLVM 值的转换，变量到 alloca + load/store
- **二值返回**：`return e1, e2` → LLVM struct 打包，调用方多接收解包为字段提取。单接收取第一字段
- **控制流**：if/else → basic blocks + phi 节点合并，for 循环 → 4 个 basic blocks（init/cond/body/exit）
- **Arena 分配**：字符串拼接和 push 操作在全局 Arena 上分配，bump pointer 递增
- **运行时检查**：数组/字符串索引越界检查 → icmp + conditional branch + call panic
- **返回值路径分析**：phi 节点合并不同路径的返回值，不可达路径的检测

**实现策略**
- 函数体用 alloca 分配局部变量栈空间
- 二值返回用 LLVM struct type 打包，调用方 extractvalue 解包
- 字符串和数组数据统一在 Arena 上分配
- 结构体字段访问用 GEP 计算偏移
- 除零和除法溢出检查插入在 sdiv/srem 之前
- 先输出文本 IR（`emit-llvm`）肉眼验证，再切到 JIT/AOT

### 6. JIT + REPL — 中复杂度 (~500 LOC)

- **JIT 引擎**：inkwell ExecutionEngine，增量编译模式
- **REPL 循环**：readline 绑定 + 多行输入检测（括号/块/未完成语句 → 续行提示 `....>`）
- **符号表持久化**：跨 REPL 输入的符号累积（函数定义、变量、结构体定义）
- **REPL 命令**：`:quit` 退出、`:show` 打印当前符号表、`:reset` 清空符号表、`:load <file>` 加载源码文件
- **错误恢复**：编译错误不影响已编译的符号，错误信息带源码位置（ariadne 格式）

### 7. AOT 编译 — 中低复杂度 (~300 LOC)

- **管线**：源码解析 → 语义检查 → LLVM IR 生成 → 优化 pass（-O1/-O2/-O3）→ 目标文件（.o）→ ld 链接 libkangrt.a + musl → 可执行文件
- **CLI 子命令**
  - `kang build <file>` — 编译为可执行文件
  - `kang run <file>` — 编译并执行
  - `kang check <file>` — 仅词法/语法/语义检查，不生成代码
  - `kang emit-llvm <file>` — 输出 LLVM IR 文本到 stdout
- **入口**：`def main() -> i32`，返回值作为进程退出码

### 8. 运行时库 — 低复杂度 (~500 LOC)

- **Arena 分配器**：全局 bump allocator，线程局部（单线程），8 字节对齐，无 free
- **内置函数**（15 个函数名，含重载共 19 个签名，封装 musl libc）
  | 函数 | 签名 | 封装 |
  |------|------|------|
  | `len` | `(s: str) -> i32` / `(a: [T]) -> i32` | strlen / array.len |
  | `push` | `(arr: [T], elem: T)` | Arena realloc + memcpy |
  | `puts` | `(s: str)` | fputs(stdout) + "\n" |
  | `print` | `(s: str)` | fputs(stdout) |
  | `eprint` | `(s: str)` | fputs(stderr) + "\n" |
  | `read_file` | `(path: str) -> (str, bool)` | fopen + fread + Arena |
  | `read_line` | `() -> (str, bool)` | fgets(stdin) |
  | `write_file` | `(path: str, content: str)` | fopen("w") + fputs |
  | `append_file` | `(path: str, content: str)` | fopen("a") + fputs |
  | `file_exists` | `(path: str) -> bool` | access(F_OK) |
  | `file_size` | `(path: str) -> (i32, bool)` | stat + st_size |
  | `str` | `(i32) -> str` / `(f64) -> str` / `(bool) -> str` | snprintf |
  | `i32` | `(s: str) -> (i32, bool)` / `(f64) -> i32` | strtol / trunc |
  | `f64` | `(s: str) -> (f64, bool)` / `(i32) -> f64` | strtod / f64 cast |
  | `bool` | `(s: str) -> (bool, bool)` | strcmp "true"/"false" |
- **panic 处理**：fprintf(stderr, ...) → exit(1)
- 函数前缀 `k_`，编译为 `libkangrt.a`

### 9. 测试 — 中复杂度 (~1,000 LOC)

- **单元测试**：词法 token 快照测试、parser 输入输出对比
- **集成测试**：.kang 源文件 → 编译执行 → 断言 stdout/stderr/exit_code
- **fuzz**：随机语法正确程序输入，验证编译器不崩溃

## 风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| LLVM IR 调试困难 | 高 | 中 | 先用 `emit-llvm` 输出文本 IR 肉眼验证 |
| Arena 内存布局出错 | 中 | 中 | 先用 C 写原型，验证 layout 再迁移 Rust |
| inkwell API 文档不足 | 中 | 低 | 对照 LLVM C++ 文档，inkwell 是 1:1 映射 |
| 类型规则实现遗漏 | 低 | 低 | 需求阶段已 4 轮审查，实现前可做最终规则走查 |
| JIT 符号表持久化 bug | 中 | 中 | 增量测试每个 REPL 输入，错误后验证符号表不污染 |
| musl 静态链接问题 | 低 | 中 | musl 成熟稳定，主要风险是 Rust FFI 的 C ABI 对齐 |

## 结论

Kang v2 是一个**中等偏上规模的学习项目**，估量约 6,500 行 Rust 代码。
核心挑战集中在 LLVM 代码生成阶段（1,800 行，27% 代码量，约 40% 工作量），其中二值返回的 struct 打包/解包和返回值路径 phi 节点合并是新增的实现复杂度。
语言设计约束严格（零 UB、显式类型、无隐式转换）降低了整体复杂度——大量不确定行为在规格阶段已被消除。
二值返回 `(T1, T2)` 替代了哨兵错误模式，调用方代码更清晰，且语法层已杜绝多返回类型的滥用（`Type` 规则不含 `(...)`，仅 `ReturnType` 可用）。
项目结构清晰，模块间耦合低（词法→语法→语义→代码生成，单向依赖），适合渐进式开发。
