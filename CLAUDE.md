# CLAUDE.md — Kang v2 项目协作规则

## 项目定位
Kang 是教学用 toy 语言，严格静态类型，支持 JIT REPL 和 AOT 编译。AI 主导实施，人类在里程碑节点审查。

## 核心准则

所有代码和文档必须满足五项原则，优先级从高到低：

1. **可理解性** — 维护者（人类）能在 10 分钟内理解任何一个模块的职责、输入输出、核心算法，不需要跟踪全局状态或隐式约束
2. **可维护性** — 修改变更隔离在单一模块内；公共接口稳定；代码结构清晰，不藏暗坑
3. **可扩展性** — 在阶段边界插入新 pass 不改动上下游；新增语言特性不改动已有模块的公共接口
4. **模块解耦** — 每个模块是纯函数 `IR_in → IR_out`；模块间通过标准化的 IR 文本或类型安全的数据结构交互
5. **最小依赖** — 不引入不必要的第三方库；模块只依赖它直接使用的下游模块，不传递依赖

**核心禁令: 绝不给维护者留下认知债务。** 任何"临时方案"必须在提交前清理干净。代码的意图必须从命名和注释中一目了然。

## 注释规范

**必须写注释的场景：**
- 每个文件开头: 一句话说明该模块的职责和它在流水线中的位置
- 每个公开函数: 一行说明它的作用、参数含义、返回值含义
- 数据结构 (struct/enum): 每个字段/variant 一行注释
- 非显而易见的算法: 解释 WHY 这样做，不是 WHAT 做（WHAT 从代码可读）
- 类型/语义规则的实现处: 标注对应的规则编号，如 `// T3: if 条件必须是 bool`

**不写注释的场景：**
- 自解释的局部变量和简单赋值
- 一眼能看懂的 getter/setter 类函数
- 标准库/框架的惯用模式（如 `impl Display for Token`）

**注释风格：**
- 使用 `//` 行注释，不使用 `/* */` 块注释（留给临时调试用）
- 中文注释，简洁直白
- 不要写"用于 X 的 Y"这种废话，写"将 X 转换为 Y 因为 Z"

## 提交规范

**必须提交的时机:**
- 任何一个逻辑单元完成并验证通过后，立即提交
- 文档更新、代码实现、测试补充各自独立提交，不混在一起
- 不允许大量未提交的修改堆积

**提交信息格式 (中文):**
```
<动词> <范围> — <一句话概述>

- <具体变更点 1>
- <具体变更点 2>
```

动词: 新增/修复/重构/更新/移除
范围: 模块名或文件名

**禁止的提交信息:**
- `update` / `fix` / `wip` 等无意义单词
- 只写"根据要求更新"而不说明具体改了什么
- 英文标题中文详情混用

## 代码风格

- 不过度抽象: 三个相似行好过一个为未来预留的间接层
- 不引入当前用不到的特性开关、配置项、兼容 shim
- 类型签名即文档: 函数签名应让调用者不需要读实现就知道怎么用
- 错误处理: 编译器阶段用 `Result<T, Error>` 传播，不做 `unwrap`/`expect`；仅在不可能是错误的地方（如构造已知合法值）允许 `unwrap`

## 模块解耦原则

- 每个编译阶段是纯函数: `IR_in → IR_out`
- AST 是 parser/semantic/codegen 之间的唯一数据契约
- kangrt 完全独立, 通过 C ABI (`extern "C" fn k_*`) 交互, 无 Rust 层面依赖
- 每个模块有独立测试集, 开发完即可验证, 不依赖后续阶段
